# Технический план

## Статус
Черновик. Основан на бизнес-контексте и функциональной карте.

## Связь с глобальной рамкой
План должен соответствовать `spec/00-глобальная-рамка.md` и описывать реализацию простого сайта, а не сложной системы.

## Цель
Сделать локальное фронтенд-приложение для управления строительными объектами в Бельгии без бэкенда, авторизации и онлайн-зависимостей.

## Что фиксируем в этом плане
- базовый стек первой версии;
- модель хранения данных;
- организацию экранов и основных модулей;
- подход к истории изменений;
- подход к локальному AI-сценарию;
- границы первой реализации.

## Предлагаемый стек
- `Vite` для быстрого запуска локального фронтенда;
- `React` для UI;
- `TypeScript` для типизации данных и сценариев;
- локальное хранение в `IndexedDB` для объектов, истории, файлов и фотографий;
- поверх `IndexedDB` тонкий слой доступа к данным;
- без серверного API на первой версии.

## Почему именно так
- проект должен работать локально;
- история изменений и вложения требуют хранения сложнее, чем один `localStorage`;
- `IndexedDB` подходит для структурированных сущностей и файлов;
- без бэкенда лучше сразу заложить простую и устойчивую локальную модель данных.

## Основные сущности
- объект;
- фирма;
- работа;
- расход;
- допработа;
- фото;
- файл;
- история изменения;
- комментарий;
- событие календаря;
- запись локального AI-анализа.

## Базовая модель объекта
- id;
- название;
- тип клиента;
- клиент;
- адрес;
- направление работ;
- статус;
- даты начала и окончания;
- примерный срок;
- финансы;
- работы;
- расходы;
- допработы;
- фото;
- файлы;
- история;
- связанные фирмы;
- риск или задержка.

## Базовая модель фирмы
- id;
- название;
- тип фирмы;
- документы;
- счета;
- связанные объекты;
- контакты и служебные данные, если они понадобятся позже.

## Базовая модель истории
- id;
- дата;
- автор изменения;
- тип сущности;
- сущность-источник;
- старое значение;
- новое значение;
- короткое описание;
- признак сохранённого AI-комментария, если запись пришла из локального анализа.

## Хранение данных
- все основные сущности хранятся локально;
- история хранится отдельно от видимых полей, но связана с объектом или фирмой;
- фото и файлы хранятся как локальные записи в `IndexedDB`;
- в интерфейсе показываются только безопасные локальные ссылки и метаданные;
- при первом запуске приложение создаёт локальную структуру данных.

## Правило записи изменений
- любое важное изменение создаёт отдельное событие истории;
- UI не должен перетирать прошлое значение без следа;
- изменение статуса, сроков, денег, работ, допработ, файлов и заметок всегда оставляет запись;
- запись истории должна формироваться в одном месте, чтобы логика была единообразной.

## Экраны первой версии
- список объектов;
- карточка объекта;
- карточка фирмы;
- календарь очередности;
- архив завершённых объектов;
- общий экран истории;
- локальный экран или панель AI-помощника.

## Поведение интерфейса
- основной навигационный слой должен быть простым;
- список объектов остаётся главным входом;
- карточка объекта раскрывает детали без лишней перегрузки;
- связанные фирмы и документы открываются внутри карточки;
- календарь даёт быстрый обзор будущей загрузки;
- архив отделяет закрытые объекты от активных.

## Локальный AI-сценарий
- AI не является онлайн-сервисом первой версии;
- пользователь запускает локальный анализ из карточки объекта;
- результат приходит как черновик текста;
- пользователь решает, сохранять его или нет;
- сохранённый черновик становится обычной записью карточки и получает событие в истории.

## Состояния и логика
- `в работе` - объект активен;
- `ожидает` - объект временно остановлен;
- `задержка` - есть риск сдвига сроков или следующего объекта;
- `завершён` - объект закрыт;
- изменения состояния синхронизируются со списком, карточкой и календарём.

## Ограничения первой версии
- без авторизации;
- без серверной синхронизации;
- без онлайн-деплоя как обязательного сценария;
- без сложных прав доступа;
- без автоматических уведомлений;
- без полноценного backend-first процесса.

## Что нужно реализовать сначала
1. Базовое хранилище и загрузка начальных данных.
2. Список объектов с фильтрами и быстрым доступом.
3. Карточка объекта с работами, финансами, фото, документами и историей.
4. Карточка фирмы и связи с объектами.
5. Календарь очередности.
6. Локальный AI-сценарий как сохранение черновика.

## Риски
- `IndexedDB` потребует аккуратного слоя доступа и миграций;
- вложения могут быстро усложнить интерфейс;
- история может распухнуть, если не держать короткий формат записи;
- для локального AI-сценария надо заранее решить, будет ли это только имитация черновика или отдельный встроенный обработчик.

## Критерий готовности первой версии
- объект можно создать, изменить и закрыть;
- изменения попадают в историю;
- фирма связывается с объектом;
- расходы, работы, фото и документы сохраняются локально;
- календарь показывает очередность;
- локальный AI-черновик можно сохранить в карточке;
- приложение работает полностью на `localhost`.
