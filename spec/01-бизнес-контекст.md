# Бизнес-контекст

## Статус
Черновик. Это главный спек проекта.

## Краткое описание
Проект нужен для управления строительными объектами в Бельгии. Пользователь хочет видеть состояние объектов, этап работ, сроки, выполненные действия, следующие задачи и историю всех обновлений.

Пользователь также должен иметь возможность добавлять новые проекты и заполнять для каждого проекта обязательные данные.

## Бизнес-сфера
- Строительство.
- Работа производится в Бельгии.
- Объекты могут находиться в разных городах и на разных этапах.
- Основные виды работ: внутренние отделочные работы, крыши и фасады.

## Главная проблема
При управлении строительными объектами легко потерять контекст:

- что уже сделано;
- что нужно сделать дальше;
- почему объект задерживается;
- какие решения уже принимались;
- какие комментарии или анализы были сделаны раньше.

## Главная цель
Создать локальный инструмент, который помогает контролировать проекты и не терять историю по объектам.

Это стартовая версия проекта. В будущем проект должен вырасти в полноценную систему, а не остаться только фронтенд-формой.

## Первый пользователь
Первым пользователем системы будет сам пользователь проекта.

Это значит, что первая версия должна быть удобна для личного контроля строительных объектов без сложной настройки ролей, доступа и команды пользователей.

## Первый тип клиентов
Сначала проект ведётся для частных клиентов.

После этого можно будет добавить отдельный режим для объектов от генеральных подрядчиков. Это два разных сценария, и их важно не смешивать в одной одинаковой форме.

## Что считается одним объектом
Один объект для меня - это один адрес и один клиент-заказчик.

Если по одному адресу есть несколько договоров, это всё равно должен быть один объект с отдельной историей и структурой внутри.

## Рабочая нагрузка
Одновременно может быть больше 4-5 объектов в работе.

Это значит, что система должна помогать быстро понимать, что уже сделано, что нужно сделать дальше и какой следующий этап по каждому объекту.

## Планирование
Пользователю нужен не только контроль текущих объектов, но и план будущих проектов на несколько месяцев вперёд.

Это нужно, чтобы заранее понимать, какие объекты будут следующими, и строить рабочий план примерно на 2-3 месяца вперёд.

## Очередность проектов
Нужен календарь очередности проектов.

В этом календаре должно быть видно:

- когда проект начинается;
- когда он примерно закончится;
- в какой очереди он стоит;
- какие проекты идут раньше, а какие позже.

## Вид календаря
Нужен комбинированный вид: календарь и список очереди вместе.

Так пользователь сможет одновременно видеть:

- сроки;
- порядок проектов;
- будущую загрузку;
- что идёт раньше и что позже.

## Что видно в календаре
Для каждого проекта в календаре должно быть видно:

- название проекта;
- дата начала;
- примерная дата завершения;
- направление работ;
- статус проекта.

Для проекта со статусом `в работе` в списке очереди должно быть видно всё ключевое сразу из списка: название, направление, даты и статус, кроме денег, чтобы не открывать карточку без необходимости.

## Статусы проекта
Для очереди и календаря нужны простые статусы:

- не начат;
- в работе;
- ожидает;
- завершён.

Статус `ожидает` не требует отдельного указания причины в календаре.

## Планирование по направлениям
Каждое направление работ должно планироваться отдельно.

Нужны отдельные разделы и отдельный график для:

- крыш;
- фасадов;
- общих строительных работ.

Это нужно, чтобы по каждому направлению была своя планировка, своя очередь и свой график работ.

## Где будет использоваться система
Пользователь будет работать в офисе и дома, через компьютер и ноутбук.

## Что важно видеть первым
В первую очередь нужно видеть:

- следующий этап;
- что нужно сделать;
- что уже было сделано;
- всё, что по объекту влияет на контроль работ.

Это относится к списку объектов, очереди и общему обзору. В карточке объекта первым идёт блок с основной информацией.

## Что важно видеть в карточке объекта
Карточка объекта должна сразу показывать:

- данные клиента и объекта;
- смету и контракт в одном разделе;
- фактуры и оплату в другом разделе;
- отчёт по работам;
- что выполнено по работе;
- фотографии объекта;
- кто какие работы делал;
- кто был на объекте;
- план работ;
- риск или проблему по проекту;
- историю изменений.

Отчёт по работам должен собирать в один блок:

- выполненные работы;
- фотографии;
- исполнителей;
- изменения, которые важны для контроля;
- данные, нужные для понимания хода работ.

Дополнительные работы должны быть видны отдельно и содержать:

- цену, если она уже согласована;
- или оценку часов, если стоимость ещё считается;
- письменное подтверждение заказчика;
- отметку, что работы нельзя начинать до подтверждения.

Если дополнительные работы могут сдвинуть сроки, это должно быть видно как риск для планировки.

В оплате должно быть видно:

- оплачено;
- не оплачено;
- что уже выставлено в фактурах;
- что ещё осталось к оплате.

## Расходы по проекту
По каждому проекту нужно отдельно указывать расходы, чтобы можно было считать выручку.

Расходы должны быть разбиты по категориям:

- материалы;
- рабочие;
- транспорт;
- непредвиденные расходы.

Для каждой категории желательно видеть сумму и, если нужно, заметку.

Все расходы должны храниться в одном разделе проекта.

В этом разделе к завершению проекта должен автоматически считаться итог по всем суммам.

В этом же разделе нужно видеть итоговую прибыль по проекту:

- доход по проекту;
- расходы;
- прибыль как разницу между доходом и расходами.

Доход по проекту должен учитывать контракт и утверждённые дополнительные работы.

Система должна предупреждать о проблемных проектах, если допработы, сроки или расходы создают риск задержки.

## Чек-ин и ЛИМОСА
Для больших строек нужен отдельный блок регистрации присутствия рабочих.

В этом блоке должно быть видно:

- кто был зарегистрирован на объекте;
- когда был сделан чек-ин;
- по какому документу или подтверждению это сделано;
- к какой стройплощадке относится регистрация;
- какая фирма или субподрядчик был отмечен.

Для бельгийской стройки этот блок нужно связывать с ЛИМОСА и регистрацией присутствия на объекте.

## Фирмы и документы
Нужно отдельно фиксировать все фирмы, с которыми ведётся работа.

Фирма - это отдельная сущность, которая связывается с объектами и хранит свои документы, счета и историю работы.

Есть два типа фирм:

- подрядчики, которым я отдаю работу;
- генеральные заказчики, у которых я беру работу.

У каждой фирмы должны храниться свои документы и данные:

- договор с фирмой;
- документы рабочих, которые фирма присылает;
- A1;
- ЛИМОСА;
- VCA;
- все выставленные этой фирмой счета;
- другие документы по этой фирме.

Это нужно хранить под каждой фирмой отдельно, чтобы быстро видеть всю историю работы с подрядчиком.

## Важное требование пользователя
История обязательна. Каждый раз, когда есть обновление, оно должно записываться. Данные не должны теряться между шагами работы над проектом и между обновлениями объектов.

В историю обязательно должны попадать:

- дополнительные работы;
- сроки;
- проблемы;
- непредвиденные расходы;
- изменения по плану работ;
- изменения, которые могут повлиять на следующего клиента.

Запись в историю должна создаваться по кнопке сохранения после заполнения нужной информации.

В историю также нужно фиксировать:

- старт работ;
- окончание работ;
- дополнительные работы после заполнения данных;
- смену плана, если она влияет на сроки или следующий объект;
- изменения риска;
- изменения по прибыли и расходам.

## Обязательные данные проекта
В каждом проекте должны быть только самые важные данные, которые помогают быстро понять, что это за объект, кто клиент, сколько по нему денег и на какой он стадии.

Для старта нужно вести два раздельных типа объектов:

- частные клиенты;
- генеральные подрядчики.

## Как я бы записывал каждый объект
Я бы делал карточку объекта в таком виде:

### 5 основных полей

1. Название проекта.
2. Данные клиента и адрес объекта.
3. Деньги: смета и контракт.
4. Отчёт по работам и фотографии.
5. Исполнители, чек-ин и план работ: кто работал, кто был на объекте, что делали, кто был субподрядчиком.

### Отдельный финансовый раздел

- фактуры;
- оплачено / не оплачено;
- остаток к оплате;
- допработы по деньгам;
- расходы на материалы;
- расходы на рабочих;
- расходы на транспорт;
- непредвиденные расходы;
- доход;
- прибыль.

### Что особенно важно для частного клиента

- смета и контракт;
- фактуры;
- оплачено и не оплачено;
- сколько уже оплачено;
- расходы на материалы;
- расходы на рабочих;
- расходы на транспорт;
- непредвиденные расходы;
- доход;
- прибыль;
- кто работал на объекте;
- кто был на объекте;
- отчёт по работам;
- фотографии;
- план работ;
- какие дополнительные работы появились;
- цена или оценка часов по дополнительным работам;
- подтверждение заказчика на дополнительные работы.

### Нижний блок карточки

- документы и файлы;
- история изменений;
- комментарии по объекту.

Пример короткой записи:

```text
Проект: Renovation Rue de Namur
Клиент: ABC Construction
Адрес: Brussels
Деньги: смета 120000, контракт 115000, фактуры 46000, оплачено 46000
Отчёт: отделка завершена частично, фото загружены
Исполнители: Pierre, Ahmed, Subcontractor X
Чек-ин: Pierre, Ahmed
План работ: согласовать материалы и продолжить монтаж
```

## Что можно добавить позже
Чтобы проект был ещё удобнее, позже можно добавить:

- номер проекта;
- контактное лицо клиента;
- телефон или email клиента;
- ссылки на фото или материалы.

## Как я бы разделил будущие режимы
- Частный клиент: фокус на адресе, клиенте, смете, оплате, исполнителях и допработах.
- Генеральный подрядчик: фокус на адресе, заказчике, договоре, приёмке, актах, счётах и чёткой отчётности.

## Простая норма структуры
Каждое изменение в проекте должно относиться только к одной из основных сущностей:

- объект;
- фирма;
- финансы;
- планирование;
- история.

## Формат первой версии
- Только фронтенд.
- Только локально на `localhost`.
- Без онлайн-доступа.
- Без бэкенда на старте.
- Без авторизации на старте.
- Разработка только через спецификации.

## Что нужно для удобства и скорости
Чтобы работать быстрее и проще, в проекте нужны:

- быстрый поиск по объектам и фирмам;
- фильтры по статусу, направлению и типу проекта;
- шаблон нового проекта, чтобы не заполнять одно и то же каждый раз;
- дублирование похожего проекта или фирмы;
- быстрые действия без лишних переходов: добавить работу, фото, расход, фирму или проект;
- закреплённые проекты сверху списка;
- отдельный архив закрытых объектов;
- единая логика для добавления новых записей.

Также нужно, чтобы файлы и фотографии можно было быстро прикреплять туда, где они нужны:

- к объекту;
- к фирме;
- к документам;
- к отчётам и работам, если это помогает контролю.

## Будущее развитие
После первой локальной версии проект может стать полноценным решением с расширенной логикой, хранением данных и дополнительными модулями.

## ИИ-помощник
Пользователь хочет использовать локальный Codex на своём компьютере как ИИ-помощника:

- делать маленькие анализы по объектам;
- создавать комментарии;
- помогать анализировать состояние строительных проектов;
- помогать уточнять спецификации перед разработкой.

На старте это не означает встроенный OpenAI API в приложение. Сначала нужно описать рабочий процесс с Codex в спеке.

## Что пока неизвестно
- Как именно пользователь хочет запускать анализ Codex по объекту.

## Следующий этап: функционал
Функционал пока не расписываем отдельным файлом. Когда бизнес-контекст станет понятным, добавим сюда следующий раздел:

- что пользователь должен уметь делать;
- какие основные пути пользователя есть;
- какие ситуации нужно учитывать;
- что входит в первую локальную версию;
- что не входит в первую версию.

2. Какие обновления обязательно писать в историю помимо уже зафиксированных: статус, этап, комментарии, сроки, ответственный, документы?
